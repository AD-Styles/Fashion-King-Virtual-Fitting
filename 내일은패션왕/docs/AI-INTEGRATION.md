# Phase 3 — AI 통합 설계 노트 (스켈레톤)

## 목표

옷 사진 한 장(또는 텍스트 프롬프트)을 입력하면 마네킹에 입을 수 있는 형태로 변환.

## 접근 방향

### 방향 1: 2D 합성 (VITON 계열)
- **모델**: IDM-VTON, OOTDiffusion, StableVITON
- **입력**: 마네킹 렌더 이미지 + 옷 이미지
- **출력**: 합성된 2D 이미지
- **장점**: 빠른 통합, 사진처럼 사실적
- **단점**: 3D 회전 안 됨 (각 각도마다 재생성, 일관성 깨짐)
- **사용처**: "정면 룩북 미리보기" 같은 별도 모드

### 방향 2: 옷 사진 → 3D 메쉬 (이 프로젝트의 메인)
- **모델**: TripoSR, Wonder3D, InstantMesh, CRM
- **입력**: 옷 사진 (배경 제거된 정면)
- **출력**: 옷 .glb (메쉬 + 텍스처)
- **장점**: 마네킹에 입혀서 360 회전 가능, Phase 2 시스템 그대로 활용
- **단점**: 메쉬 품질이 들쭉날쭉, 마네킹 본에 자동 binding 어려움

### 방향 3: 텍스처만 생성
- **모델**: 옷 카테고리별로 기성 .glb 메쉬 보유 + AI는 텍스처(UV map)만 생성
- **입력**: 텍스트("플로럴 패턴 흰색 셔츠") 또는 패턴 이미지
- **출력**: UV 텍스처 이미지 (PNG)
- **장점**: 메쉬 품질 안정적, 360 회전 자연스러움
- **단점**: 옷의 실루엣(셔츠/티/후드/조끼)을 미리 만들어둬야 함

**MVP 추천**: 방향 3 (텍스처 생성) → 사용자 가치 빠르게, 메쉬 처리 부담 작음.

## API 아키텍처

```
[클라이언트]
  ├ 옷 사진/프롬프트 업로드
  ▼
[Next.js API Route]  /api/generate-clothing
  ├ 입력 검증, 사용자 인증
  ├ 외부 AI API 호출 (Replicate / Hugging Face / 자체 GPU)
  ▼
[AI 모델]
  ├ 텍스처/메쉬 생성 (수 초 ~ 수십 초)
  ▼
[저장]
  ├ 결과를 S3/R2/Supabase Storage에 업로드
  └ DB에 메타데이터 저장
  ▼
[클라이언트]
  └ 새 Outfit으로 자동 적용
```

## 비용/지연 고려

- **Replicate**: 모델별 초당 과금. TripoSR ≈ $0.001/sec, 1장 ~30초 = $0.03
- **HF Inference Endpoints**: 시간당 GPU 임대. 사용량 적으면 비쌈
- **자체 호스팅 (RunPod 등)**: 트래픽 많아지면 유리, 운영 부담

MVP는 Replicate가 가장 빠른 시작.

## 입력 전처리

옷 사진을 그대로 보내지 말고 전처리:
1. **배경 제거**: rembg, Remove.bg API
2. **정면 정렬**: 옷이 비스듬하면 자동 정렬
3. **사이즈 정규화**: 1024x1024 또는 모델이 요구하는 크기

## 큐와 폴링

생성에 시간이 걸리므로:
- API 라우트에서 즉시 응답 안 함 → 큐에 작업 넣고 `jobId` 반환
- 클라이언트가 `/api/jobs/:id`로 폴링 또는 WebSocket
- 또는 Server-Sent Events

작은 MVP는 단순 동기 호출로 시작해도 OK (단일 사용자 데모용).

## 보안/저작권

- 사용자가 업로드한 옷 사진의 저작권 — 본인 소유 / Fair use 명시
- 생성된 모델의 라이선스 — Replicate 모델별 약관 확인
- 사진 저장 시 EXIF 메타데이터 (위치 등) 제거
